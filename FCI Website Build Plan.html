Phase 4 — Website Build

Website Build Plan

A gated, research-backed methodology for building the FCI website. One component at a time, one page at a time, with approval gates that prevent cascading rework. Every decision here is grounded in established industry practice and our Honest Code principles.

The Problem This Plan Solves

Why Website Projects Fail

Research from CommonPlaces, Armia Systems, and two decades of post-mortem literature identify the same pattern: a team builds 15 pages based on a template that felt right on page one. By page eight, someone notices the spacing is off, the nav doesn't work on mobile, or the color system drifts. Now the fix has to be applied retroactively to every page already built. Each fix introduces new inconsistencies. The project spirals into a cycle of rework that feels productive but is actually just catching up to where it should have been.

The technical term is design drift. The emotional term is what you described: "a monster."

This plan is engineered to make that impossible. We build one page — a single, complete, pixel-perfect page — and we don't touch page two until that foundation is locked. Every visual decision, every spacing value, every color, every component exists in exactly one place. Change it there, it changes everywhere. That's not a feature request — it's the architecture.

Sources: CommonPlaces, "Top 10 Reasons Why Website Projects Fail" · Armia Systems, "Why Do Some Web Projects Fail?" · Brad Frost, Atomic Design (2016)
Three Strategies That Prevent the Monster

Strategy 1 — Design Tokens: One Source of Truth

Every color, every font size, every spacing value, every border radius, every shadow lives in CSS custom properties declared once in a single file (base.css). The components never hardcode a color — they reference var(--blue), var(--text2), var(--spacing-lg). This is the same system already working in our Brand Hub pages.

When we decide the primary blue should be slightly different, or the section padding needs adjusting, we change one line. Every page, every component, every fragment inherits the change automatically. There is no "apply it to all pages" step because there are no per-page values to update.

This is not optional — it is Honest Code's CSS convention (custom properties in :root, components reference tokens via var(--token-name), no component owns color or spacing values directly).

Source: Penpot, "The Developer's Guide to Design Tokens and CSS Variables" · Builder.io, "Design Tokens Docs" · Honest Code Architecture, CSS Conventions

Strategy 2 — Atomic Design: Build From the Bottom Up

Brad Frost's Atomic Design methodology gives us the exact sequence. We don't start by designing a page. We start by designing the smallest pieces and composing upward:

Atoms are the irreducible elements: a button, a heading, an input field, a badge, a link style. They inherit from the design tokens. Molecules are small groupings: a search bar (input + button), a nav link with an icon, a card header (eyebrow + title + subtitle). Organisms are recognizable sections: the site header (logo + nav + CTA button), a hero section, a feature grid, a footer. Templates are page layouts that arrange organisms into a structure without real content. Pages are templates filled with actual FCI content.

The discipline is: we don't build organisms until we've approved the atoms and molecules that compose them. A hero section is just a heading atom + a paragraph atom + a button atom + a background treatment. If the atoms are right, the hero is right. If we skip atoms and go straight to "build the homepage," we're guessing at details and will pay for it later.

Source: Brad Frost, Atomic Design (atomicdesign.bradfrost.com) · Design Systems, "Build Systems, Not Pages"

Strategy 3 — Phase Gates: No Forward Motion Without Approval

The phase-gate process (also called stage-gate) comes from product development methodology and has been adapted across the entire software industry. The core idea: the project is divided into discrete phases, and between each phase is a gate — a formal approval checkpoint. You do not pass through the gate until the deliverables for that phase are reviewed and accepted.

For our website, this means: you see and approve the design tokens before I build a single component. You see and approve the components before I build a single page. You see and approve the homepage before I build page two. At no point are we building ahead of what's been validated.

This is the opposite of "build everything, then review." It's also the opposite of "build one page and hope it generalizes." It's a structured escalation from the smallest decisions to the largest, with your sign-off at each level.

Source: Wikipedia, "Phase-Gate Process" · ProjectManager.com, "Stage-Gate Guide" · Robert G. Cooper, "Winning at New Products" (2017)
How This Maps to Honest Code

The Architecture Is the Strategy

Honest Code isn't just how we write Python — it's the philosophy that prevents the monster. Here's the direct mapping:

P06 DOM as State (DATAOS) — The server renders complete HTML. No client-side framework building a virtual DOM that drifts from the server's view of reality. The HTML you see in the browser is the truth. HTMX swaps fragments; it doesn't rebuild the world.

P07 HTML Attributes Over Imperative DOM — hx-get, hx-target, hx-swap replace hundreds of lines of JavaScript event wiring. Less code means fewer places for things to break.

P04 I/O at the Boundary — Route handlers do I/O and return templates. Pure functions in app/logic/ do the thinking. The template layer has no surprises because it receives clean data.

P15 Declarative Equivalents Over Lifecycle Hooks — hx-trigger="load" replaces componentDidMount. The page arrives ready. No initialization sequence. No loading spinners hiding architectural debt.

CSS Conventions — Custom properties in :root, BEM naming, one file per component. This is the design token strategy encoded as a rule, not a suggestion.

These principles mean the website is structurally incapable of the problems that cause cascading rework. The design system lives in tokens. The components reference tokens. The pages compose components. Change flows downward, never sideways.

The Seven Gates

How gates work: Each gate produces a deliverable you can see and interact with. You review it. If something is wrong, we fix it at that level — where the fix is small and contained. Only when you say "approved" do we proceed. Nothing built at Gate N+1 ever contradicts what was approved at Gate N, because Gate N+1 is composed from Gate N's outputs.

0
Project Skeleton & Design Tokens
The foundation everything else inherits from

What I Build

The FastAPI project structure per Honest Code architecture: app/main.py, app/routes/, app/logic/, app/templates/, app/static/css/. The base.html shell with the HTMX script tag. And critically: base.css with every design token — every color, font size, font weight, spacing value, border radius, shadow, transition duration, and breakpoint — declared as CSS custom properties in :root.

I also build the dark/light theme system (already proven in our hub pages) and set the :root overrides for each theme.

What You Review

The complete design token list — every color, spacing, and type value — presented as a visual reference sheet
The project folder structure matches Honest Code architecture
base.html renders a blank page with correct fonts, background, and theme toggle
Dark and light mode switch correctly

Why This Gate Exists

If the tokens are wrong, every single component, page, and interaction built on top of them will be wrong. This is the cheapest possible time to debate whether --spacing-section should be 64px or 80px. After this gate, that question is answered forever.

Gate 0 Approval → "The design tokens and project structure are correct. Proceed to components."

1
Atoms & Molecules — The Component Library
Every UI element the site will ever use, built and tested in isolation

What I Build

Each component as its own BEM-named CSS file in static/css/components/ and its own Jinja2 fragment. Built in isolation on a dedicated "component gallery" page (like a Storybook, but as a simple HTML page within our own app — no external tooling needed).

Atoms: Button (primary, secondary, ghost variants) · Heading (h1 through h4 with Inter Tight weights) · Body text (paragraph, small, caption) · Link (inline, standalone, nav) · Badge/eyebrow · Input field · Textarea · Form label · Divider/rule · Icon container · Logo mark

Molecules: Nav link with dropdown indicator · Card (icon + title + description) · CTA block (heading + paragraph + button) · Form group (label + input + error message) · Stat block (number + label) · Team member card (photo + name + title) · Article preview (date + title + excerpt) · Breadcrumb trail

What You Review

The component gallery page showing every atom and molecule in both dark and light mode
Each component at mobile (375px), tablet (768px), and desktop (1280px) widths
Hover states, focus states, and active states on interactive elements
Typography feels right — the sizes, weights, spacing, and line heights feel like FCI

Why This Gate Exists

These atoms and molecules are what pages are made of. A homepage hero is just: heading atom + paragraph atom + CTA molecule + background treatment. A solutions page is just: heading + card molecules in a grid + CTA block. If the building blocks are right, the pages assemble correctly. If we skip this step and go straight to pages, we're making atomic-level decisions inside page-level contexts, which means every page will make them slightly differently.

Gate 1 Approval → "The components look right, feel right, and work at all sizes. Proceed to navigation."

2
Navigation & Page Shell — The Organism Layer
Header, footer, and navigation tested as a standalone experience

What I Build

The site header organism: FCI logo + primary nav with dropdowns (Markets, Solutions, Resources, About) + Contact CTA button. The footer organism: three-column link structure + bottom bar with legal links and phone numbers. The mobile hamburger menu with animated open/close. All built as Jinja2 includes inside base.html.

At this gate, the page body between header and footer is empty (or contains placeholder text). The entire focus is: does the navigation work? Does it feel right? Does the dropdown open and close correctly? Does the mobile menu work? Does the sticky header behave properly on scroll?

What You Review

Desktop nav: hover states, dropdown menus, active page indicator
Mobile nav: hamburger icon, slide-out or overlay menu, all links accessible
Footer: all links present, phone numbers correct, layout balanced
Sticky header behavior on scroll — does it feel premium and not janky?
Navigation tested on an actual phone (or phone-sized browser window)

Why This Gate Exists

Navigation is seen on every single page. If the nav is 2px off, or the dropdown feels laggy, or the mobile menu doesn't close properly, that bug exists on all 24 pages simultaneously. Testing it in isolation — with nothing else on the page to distract — means we see exactly how it behaves. Fixing a nav bug here is a 1-file change. Fixing it after 24 pages are built is 24 pages of regression testing.

Gate 2 Approval → "The navigation is pixel-perfect, works on mobile, and the header/footer are approved. Proceed to the first page."

3
The Homepage — The Proof of Concept
One complete page that proves the entire system works

What I Build

The full homepage, using only the approved components from Gates 0–2, filled with the approved Website Copy content. This includes: hero section with the FCI positioning statement, the "what we do" overview, the market segments with links to broker-dealers & branch offices / insurance carriers & agencies / RIAs, the solutions preview, social proof / authority signals, and the CTA section.

This is the single most important page on the site. It's also the hardest — it uses the most components, has the most visual variety, and sets the standard for everything that follows.

What You Review

Does the page feel like FCI? Does it match the brand direction from our Hero explorations?
Content hierarchy: does the eye move where it should?
Mobile rendering: does it read well on a phone?
Performance: Lighthouse score (targeting 90+ on Performance, 100 on Accessibility)
Load time: the page should render in under 2 seconds on a 4G connection
All links in nav point to real routes (they can be empty pages — but no 404s)

Why This Gate Exists

This is the proof that the system works. If the homepage looks right using only the approved tokens and components, then every other page will look right too — because they're made of the same pieces. If something feels wrong here, we trace it back: is it a token problem (Gate 0), a component problem (Gate 1), a nav problem (Gate 2), or a layout problem (this gate)? We fix it at the right level. We never "patch" a page.

Gate 3 Approval → "The homepage is right. The design system is proven. Proceed to build the remaining static pages."

4
All Static Pages — The Rollout
15 "Ready" pages built from the proven system

What I Build

All pages with "Ready" content status, in this order: About FCI, What Changes, the three Market pages (broker-dealers & branch offices, insurance carriers & agencies, RIAs), the five Solution pages (Managed Cybersecurity, Zero Trust, CISO Portal, Compliance & Exam Readiness, Security Assessment), Team & Leadership, Careers, Speaking & Media. Each page is a Jinja2 template extending base.html, composed entirely from approved components.

Because these pages are built from the same atoms and molecules approved at Gate 1, and laid out using the same grid and spacing tokens approved at Gate 0, they should assemble quickly and consistently. This is where the upfront investment pays off.

What You Review

Each page reviewed individually — does the content feel right in this layout?
Cross-page consistency: do the pages feel like they belong to the same site?
Internal links work: clicking from one page to another navigates correctly
Mobile spot-check: pick 3 pages at random, check them on mobile

Why This Gate Exists

This is a volume gate — 15 pages is a lot of content. But because the components and tokens are locked, the work here is layout and content placement, not design decisions. If we discover a problem that affects all pages, it's almost certainly a token or component issue, which means fixing it once fixes it everywhere. That's the whole point.

Gate 4 Approval → "All static pages look consistent and correct. Proceed to dynamic content."

5
CMS, Forms & Dynamic Content
News, Cyber Bites, Contact form, and HTMX interactions

What I Build

The dynamic layer: SQLite database for News articles and Cyber Bites (seeded with the 40 migrated items each). Listing pages with HTMX-powered pagination and filtering — no full page reloads, just fragment swaps. The Contact form with server-side validation (P11 — type declarations over imperative validation). The admin CMS interface for creating and publishing content, behind authentication.

This is the first gate where we add real interactivity beyond navigation. The HTMX approach means the server returns HTML fragments, not JSON. The browser swaps them in. No client-side state to manage, no API contracts to maintain (P06 — DOM as State).

What You Review

News listing: articles load, pagination works, clicking an article shows the full post
Cyber Bites listing: same as News
Contact form: submit with valid data (confirm it stores), submit with invalid data (confirm it shows errors)
Admin CMS: create a test article, edit it, publish it, confirm it appears on the public site
HTMX interactions feel smooth — no flashing, no layout shift during fragment swaps

Why This Gate Exists

Dynamic content introduces a database, server-side logic, and HTMX interactivity. These are new failure modes that don't exist in static pages. Testing them in isolation — after the static pages are proven — means any bug we find is definitively in the dynamic layer, not in the design system.

Gate 5 Approval → "Dynamic content works correctly. The CMS is functional. Proceed to launch preparation."

6
Launch Readiness — Performance, SEO & Security
The final quality gate before DNS cutover

What I Build

301 redirects from every old WordPress URL to its new equivalent (preserving SEO equity). Meta tags, Open Graph tags, and structured data on every page. Security headers (CSP, HSTS, X-Frame-Options). Sitemap.xml and robots.txt. Performance optimization: image compression, CSS minification, preconnect hints, proper caching headers. Accessibility audit: every page scores 100 on Lighthouse accessibility.

What You Review

Lighthouse report: Performance 90+, Accessibility 100, Best Practices 90+, SEO 90+
Every old fcicyber.com URL redirects to the correct new page (no 404s for existing links)
Share a page on LinkedIn — does the preview card show the right image and description?
Full mobile walkthrough: navigate the entire site on a phone from homepage to contact
Content review: final read-through of every page for typos, outdated info, broken links

Why This Gate Exists

Once DNS is cut over, the old WordPress site is gone. There is no "go back." This gate ensures we've caught everything before the irreversible step. The redirect map is especially critical — every existing link to fcicyber.com from LinkedIn posts, email signatures, regulatory filings, and partner sites must land on the right page.

Gate 6 Approval → "The site is ready. Cut over DNS."

How I Will Work

My Commitment to This Process

I will not skip gates. Even if I'm confident a component is right, I show it to you before building on top of it. Confidence without verification is how design drift starts.

I will not build ahead. If Gate 2 (navigation) is under review and I have time, I don't "get a head start" on Gate 3 (homepage). Building ahead creates pressure to approve things that aren't ready, because work has already been done on top of them.

I will commit and push after every change. Per our workflow, no batching. Every change is individually versioned. If something goes wrong, we can revert to the exact commit before the problem — not "somewhere last Thursday."

I will test my own work before declaring it done. Every gate deliverable will have been tested at mobile, tablet, and desktop widths, in both dark and light mode, before I show it to you. You should never be the first person to catch a responsive issue.

I will trace bugs to the right level. If a page looks wrong, the question is: is the token wrong (Gate 0)? Is the component wrong (Gate 1)? Is the layout wrong (this gate)? I fix at the root, not at the symptom. A "page fix" that changes a token value is a design system fix — it gets tested as one.

How Each Work Session Will Flow

Step 1: Pull. Always start by pulling the latest from the repo. No working on stale code.

Step 2: State the scope. "Today I'm working on Gate 1 — building the button atom and card molecule." You know exactly what I'm doing before I do it.

Step 3: Build. Write the code, the CSS, and the template. Follow Honest Code principles — pure functions in logic/, I/O at the boundary in routes/, design tokens in base.css, BEM components in their own files.

Step 4: Self-test. Check at 375px, 768px, 1280px. Check dark mode. Check light mode. Check hover states. Check focus states for keyboard navigation.

Step 5: Commit and push. One commit per logical change. Clear commit message. Push immediately.

Step 6: Present for review. Show you the deliverable with a link to the live page. State what to look for. Wait for approval before proceeding.

Performance Budget

Measurable Performance Targets

These are not aspirations — they are pass/fail criteria checked at every gate. If a gate's deliverable fails a target, it does not pass.

Largest Contentful Paint (LCP): under 2.5 seconds. This is the time until the largest visible element renders. Our stack advantage: server-rendered HTML arrives ready. No JavaScript bundle to parse before the first paint.

Interaction to Next Paint (INP): under 200ms. This is the time between a user interaction and the browser's response. HTMX fragment swaps are inherently fast because they're small HTML payloads, not full-page re-renders.

Cumulative Layout Shift (CLS): under 0.1. No content should jump around after the page loads. This means: explicit width/height on images, no late-loading fonts that reflow text, no HTMX swaps that change the layout above the current scroll position.

Lighthouse Performance: 90+ on every page. Lighthouse Accessibility: 100 on every page. Total page weight: under 500KB for any page (HTML + CSS + images). No external JavaScript frameworks beyond HTMX (~14KB gzipped).

Source: Google, "Core Web Vitals" · DebugBear, "Working with Performance Budgets" · web.dev, "Largest Contentful Paint"
